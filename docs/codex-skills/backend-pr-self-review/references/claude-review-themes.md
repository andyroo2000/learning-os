# Claude Review Themes for Backend API PRs

This reference summarizes recurring Claude feedback from recent `learning-os` backend PRs. Treat these as pre-push review rules, not style preferences.

## Validated Data Is the Source of Truth

- After a FormRequest validates data, access request values from `$this->validated()` instead of raw accessors such as `$this->input()`, `$this->has()`, `$this->integer()`, or `$request->query()`.
- Keep FormRequest `authorize()` aligned with Laravel's authorization contract. Authentication/401 behavior should usually come from route middleware; throw `AuthenticationException` from `authorize()` only when the route intentionally lacks middleware coverage and the 401 path is documented and tested.
- Preserve the distinction between omitted values and explicit nullable values. Use `array_key_exists` when `null` is meaningful; use `??` only when omission and `null` should behave the same.
- For PATCH/partial-update DTOs, make `has*` or presence flags the first gate for that field. If `hasImagePrompt` or `hasPreviewAudio` is false, oversized/malformed companion input should not be normalized or rejected because the caller explicitly chose not to touch it; cover every gated field type, including strings, JSON payloads, and media refs, for direct callers as well as HTTP.
- Laravel's `integer` validation rule does not cast query-string values. Cast validated integers before passing them into actions, filters, cursors, or pagination logic.
- Add tests for malformed array-shaped query/body inputs such as `?per_page[]=10` whenever a raw accessor previously existed or could accidentally coerce the value.
- For conditional validation such as `required_without`, `required_if`, `exclude_with`, or discriminator-derived fields, cover both stale-client payloads and the normal new-client shape where the excluded field is absent.
- Keep enum validation messages and accepted-value lists derived from the enum or a shared helper when possible. Hardcoded lists drift silently when enum cases change.
- Keep invalid-enum messages distinct from cross-field consistency messages unless legacy compatibility requires the same text. If a ConvoLab-compatible message intentionally conflates malformed values, missing payloads, or mismatches, add a short durable comment and a test that locks that client-visible wording.
- Keep missing-field, malformed-type, list-versus-object, and cross-field messages distinct when clients can act on the difference. Reusing one compatibility string for structurally different failures is only acceptable when legacy clients require it and tests lock the wording for each shape.
- When one operation has both idempotent same-ID retry and conflicting different-ID outcomes, make the conflict message specific enough that clients can tell which path happened.
- When HTTP validation has specific messages but direct DTO/action validation can fail for the same payload family, avoid collapsing every direct-caller failure into a generic domain error. Prefer field-aware factories or at least targeted exceptions for likely mistakes such as media-kind/source mismatches.
- For text column length guards in actions/DTOs, use character-aware length checks such as `mb_strlen` rather than byte-oriented `strlen` unless the database contract is explicitly bytes. Confirm named constants describe the same unit as the DB column and HTTP rule, especially under multibyte encodings such as utf8mb4; when character-vs-byte interpretation is easy to confuse, cover multibyte exact-`MAX` acceptance and multibyte `MAX + 1` rejection, and include a 4-byte/supplementary-code-point case such as emoji when mobile/user metadata can contain it. Cover both `MAX + 1` rejection and exact `MAX` acceptance for strict column boundaries.
- When direct action guards mirror FormRequest validation, keep guard ordering deliberate. If blank checks and length checks can both fail across several fields, align the first surfaced error with the HTTP layer or document why one class of violation intentionally wins.
- When a validation test targets one field in a cross-field payload, pass valid companion fields needed to avoid earlier rules and comment why they are present. If companions are unnecessary, omit them consistently so data providers do not imply hidden dependencies.
- Keep repeated client-visible validation strings in one place, especially when the same message is returned from both `messages()` and `after()` callbacks. Use a private constant or helper so compatibility wording cannot drift between rule-based and custom validation paths.
- When a request relies on shared validation-message helpers, verify every rule key the new request can trigger, such as `field.present`, `field.required`, and `field.array`. Test null, scalar, list-shaped, and per-attribute variants so a passing string case does not hide a nullable path falling back to Laravel's default message.
- For optional nullable fields with defaults, cover both explicit `null` and omitted-key behavior when clients can rely on both shapes. Incidental coverage through unrelated error/rate-limit tests is weaker than a focused contract assertion.
- For partial updates, explicitly test the difference between omitted fields preserving existing state and present `null` fields clearing/defaulting that state. This is especially important for enum/default fields such as media placement where `null` maps to a concrete default.
- At DTO/direct-caller boundaries, do not use `?? []` or similar defaults to make a present-but-null required payload pass shape validation unless null-clearing is the intended contract. If HTTP rejects `field: null` for a present field, the DTO/action should usually reject `hasField: true, field: null` too.
- Decide whether an empty PATCH body is a valid no-op or a validation error. If autosave intentionally returns 200 with current state and no dirty write, cover or comment that behavior so future readers do not mistake it for missing validation. For valid no-ops, prefer an early return before transactions or locks unless the endpoint intentionally needs a fresh state recheck, and explicitly state whether status guards like `Generating` still apply to pure readback.
- Inside FormRequest `after()` callbacks, prefer the callback's `$validator` safe data or captured already-normalized data over calling `$this->validated()`. Re-entering validated-data logic inside validation callbacks is harder to reason about and can obscure which rule phase owns the value.
- In `after()` cross-field checks, treat a sibling field with existing validation errors as invalid even if raw input is present and non-null. Otherwise combined error responses can falsely skip the dependent-field error for malformed sibling payloads.
- Pick one normalization/defaulting contract for request accessors, helpers, and value-object methods: either accept raw input and normalize/default inside, or require pre-normalized input and trust callers. Avoid double-normalizing/defaulting in requests plus DTOs/actions unless the redundancy protects direct callers and is documented/tested.
- When a shared validation trait has an optional mode that skips assigning cached properties, initialize those properties to safe nullable defaults or make the accessors explicitly unavailable. Future callers should not get uninitialized-property errors merely by using a supported relaxed mode.
- For JSON request fields, validate shape as well as type once the client contract is known. If a discriminator such as `creation_kind` implies different valid JSON structures, cover each shape, malformed shape, and stale-client mismatch before the API writes to storage.
- Add size/depth limits for client-provided JSON before it enters sync or retry paths. Plain `array` validation can accept deeply nested or very large payloads that are expensive to canonicalize or serialize.
- If a JSON byte cap uses a custom `json_encode` option set, compare it to the serialization that storage/model casts actually use or document that it is a soft pre-storage estimate. Tight caps should measure the authoritative representation, not a prettier or more compact proxy.
- When HTTP and action layers intentionally enforce the same JSON limit, avoid duplicated magic constants or add a focused equality/contract test. A mirrored guard is good defense-in-depth, but silent drift between layers creates incompatible direct-caller and HTTP behavior.
- Sharing constants is not enough when HTTP and DTO/action layers copy the same nontrivial validation algorithm. Recursive JSON depth/size checks, media-reference shape checks, and similar rules should usually live in a pure validator or value object that both FormRequests and direct-caller DTOs delegate to, keeping HTTP message mapping at the request edge.
- When DTOs/actions accept the same JSON arrays as a FormRequest, mirror shape/depth limits at that boundary or document why the action is HTTP-only. Direct callers should not bypass ConvoLab-compatible payload guards merely because HTTP validation caught them first.
- For shared JSON guards, cover each distinct client-visible guard through HTTP and direct action tests when both entrypoints exist. A depth test through HTTP plus a size test only at the action layer still leaves the compatibility response shape unproven.

## Client-Visible ULID Normalization

- Normalize client-provided ULIDs before `ulid` validation in FormRequest `prepareForValidation()`.
- Normalize again at action or DTO boundaries when direct callers can bypass HTTP requests. If this double normalization looks redundant, leave a short comment explaining the caller contract.
- When a malformed client ID is expected to return no result instead of throwing, make the normalizer contract explicit with a test or a local try/catch/upfront validity check. If direct action callers can pass arbitrary strings, prefer a cheap format guard before querying so non-ULID garbage does not waste a DB round-trip; if HTTP validation is the only public boundary, document that assumption. When a route constraint such as `whereUlid()` is expected to make a downstream canonicalizer infallible, confirm both accept the same input space or add a focused HTTP/action test so an unexpected `CanonicalUlid::normalize()` exception cannot surface as a 500.
- For lookup/show actions with format guards, cover malformed IDs and valid-but-missing IDs as separate paths. A deleted/cross-user HTTP 404 test is useful, but it should not be the only proof that a direct valid ID with no matching row returns the intended null/not-found result.
- Preserve non-string invalid inputs. Arrays, objects, omitted optional values, and whitespace-only values should still reach validation or explicit guards in a way that produces the expected error.
- Do not merge optional IDs as `null` when the client omitted the field. Omission and explicit null are different when validation rules or action contracts depend on presence.
- Check adjacent client-visible IDs before pushing: `id`, `card_id`, `course_id`, nested batch IDs, route params, deck/card/media/review event fields, and list filters.

## Middleware-Disabled Normalization Tests

- When a FormRequest owns trimming or lowercasing, add focused tests with `withoutMiddleware(TrimStrings::class)`.
- Cover trim-only, lowercase-only, and combined padded-uppercase input when each can regress independently.
- Keep at least one full production middleware-stack test for uppercase or padded input. Do not replace stronger existing coverage with a narrower middleware-disabled test.
- For blank or whitespace-only values without `TrimStrings`, assert the expected 422 boundary behavior unless the action is intentionally responsible for rejecting blanks.

## Layer and Domain Coverage

- If a request and action both accept the same value, test both the HTTP path and direct action path.
- When a pattern changes in one domain, inspect sibling domains before pushing: courses, decks, flashcards/cards, media assets, review events, sync feeds, and auth/user isolation.
- For list filters, include happy path, malformed ULID, blank/whitespace, cross-user isolation, pagination query preservation, and any deleted-resource behavior relevant to the endpoint.
- For batch payloads, preserve item indexes and non-array item behavior so validation errors point to paths such as `events.0.id`.
- Server-owned fields should not be mass-assignable on the model just because request/action layers currently strip them. Use explicit assignment or a dedicated domain method, and add a direct model-boundary test when the ownership contract matters.
- When testing process-owned fillability, assert every protected field named in the PR or model contract, including ownership fields such as `user_id`, derived fields, status fields, and error/retry metadata.
- If one persisted field is derived from another, such as `card_type` from `creation_kind`, default to making the derived field process-owned: remove it from fillable input, derive it from the canonical field, and test direct model creation/update. Use a caller-supplied consistency guard only when the PR documents why derivation is not appropriate.
- If an action validates a persisted field for consistency, prefer assigning that field in the action so the invariant is self-contained. Rely on model hooks/accessors for derivation only when the action does not own the input or the PR explicitly documents why central derivation is required; tests should inspect stored attributes when persistence is the contract.
- Shared domain limits and accepted-value lists should live at the domain/model/support level, not copied across Create DTOs, Update DTOs, and FormRequests. When create and update endpoints intentionally share caps such as media sources or prompt lengths, use one named constant/helper so future changes cannot drift by layer. Put wire-format or serialization limits, such as JSON byte/depth caps, media-reference allowed keys, and source strings, in a support validator/value object when they describe client payload shape more than persistence.
- If a shared constant or enum-like list must stay in sync with migration constraints, casts, serializers, or another domain's accepted values, leave a narrow doc comment or test that points future edits to the mirror location.
- For edits or retries of records in an error/status state, decide whether user changes preserve or clear server-owned error metadata and generated outputs. Preserve old `error_message`, preview payloads, generated media JSON, role markers, placement/role markers, or other nullable output-like fields only when that is intentional and covered or commented; clear them in the same action when the edit/retry semantically means stale failed output should disappear. Before adding ambiguous fields such as `*_role`, placement, prompt, or source selectors to a reset list, confirm whether they are server output or user intent. If preserving user-owned content is part of the retry/reset contract, assert it at the action/domain layer as well as through any API resource shape. If a reset list is likely to grow with the schema, prefer a named model/domain helper such as `resetGenerationOutput()` or a central field list over repeated action-local assignments and comments. Keep helper visibility as narrow as the invariant allows; a reset helper that is only safe as part of a status transition should be private behind the transition method, not a public way to clear outputs without changing state. If a public model transition helper is kept for local convention or direct unit tests, note that callers bypass action-level guards such as locks, ownership, committed checks, or allowed-status checks before adding more call sites.
- If a model hook derives a persisted field on `saving`, avoid dirtying the derived field on unrelated updates. For existing models, put the "source field is clean" early return before validation that is only needed for derivation, and test a pure status/metadata save if enum casts or raw/original comparisons can make dirty tracking noisy.
- Test intentional model-invariant exceptions at the model boundary. If a `saving` hook throws when a required source field is missing on create, add a focused test so future lifecycle changes do not turn the invariant into an accidental 500 path; make sure the application guard wins before a NOT NULL or FK constraint if the exception shape matters.
- Test explicit programming guards on action inputs when a PR introduces them, even if controllers normally make the branch unreachable. Examples include non-positive user IDs, missing persisted model IDs, or invalid direct-caller preconditions that throw `LogicException`. Match the test data to the named boundary: a "non-positive" guard should cover both zero and a negative value, not only `0`; prefer a data provider or separate tests over manual try/catch loops so each case has clear failure attribution. When targeting one guard, pass valid values for unrelated arguments so future validation-order changes cannot make the test pass for the wrong reason. Compare sibling actions so one CRUD action does not grow a defensive direct-caller guard while create/update/delete siblings silently omit or intentionally avoid it.
- When a model helper's contract is wider than the database constraint, name that boundary. For example, if a NOT NULL enum source field can still be set to `null` in memory, decide whether the helper should throw, ignore unchanged updates, or rely on validation before future update actions call `save()`.
- For lifecycle/status gates, prefer an allowlist of statuses that may proceed over a denylist of currently-blocked statuses when future enum cases could be unsafe by default. If a denylist is intentional, comment why new statuses should inherit the allowed behavior. For "all statuses except X" tests, derive the provider from the enum cases or add an exhaustiveness assertion so a future `pending`, `cancelled`, or terminal case cannot bypass coverage silently.
- When multiple terminal or lifecycle guards can overlap, choose the response precedence deliberately. If a committed/deleted/terminal guard runs before a status guard, make the reason visible near the action when the combination is possible or when the guard is purely defensive.
- Check Eloquent event ordering when multiple hooks mutate or validate the same model. `saving` runs before `creating`/`updating`, so a later exception can leave in-memory attributes changed even though nothing persisted; callers that catch such exceptions should not reuse the tainted model instance.
- Factories should not set process-owned or hook-derived fields that the model always overwrites. Dead factory values make the contract harder to see and can hide regressions where the hook stops running.
- For immutable model fields guarded by Eloquent events, choose `creating`, `updating`, or both deliberately. Test that initial server assignment is allowed and later mutation is rejected, so lifecycle-event behavior is not left implicit.
- When a controller maps several domain exception outcomes, test every branch that changes client-visible status or secrecy: cross-user conflicts should stay hidden, deleted-resource conflicts should use the intended gone/conflict shape, and normal conflicts should remain distinct.
- When a domain exception family maps named validation outcomes back to request fields, test each field-producing branch through the HTTP layer. For paired fields such as front/back, prompt/answer, or role/media, do not let the first missing side mask the second branch; construct one case where the first side is valid and the second side fails.
- When tests assert generated validation/error strings, prefer constants, exception factories, or the validator's own message source over duplicated string literals. Hardcoded copies make wording changes look like behavior failures.
- User-scoped write actions should make their ownership boundary explicit. Prefer accepting the authenticated user ID/principal separately from the target model, or document and test that direct callers must pass a model already scoped to the intended user; re-querying by `$model->user_id` only proves the model exists for its owner, not that the caller is that owner.
- Avoid blind `(int)` casts of `$request->user()->getAuthIdentifier()` at controller boundaries. Prefer an existing integer user ID property/helper, validate the identifier shape before casting, or document that the guard is configured for integer IDs; otherwise a future string/UUID guard can silently become `0` and surface as a 500 from a downstream domain guard. If a shared resolver deliberately uses `getRawOriginal()` to avoid Eloquent key coercion, do not fall back to `getAttribute()` unless the fallback is proven safe and directly tested; otherwise the fallback can re-enter casts and defeat the malformed-ID guard. Validate the raw value's type before casting, accepting only integers and digit-only strings unless the app explicitly supports another key type; reject floats, booleans, arrays, and objects rather than letting `(int)` coerce them. Cover every resolver branch, including unsynced/new model attributes where `getRawOriginal()` returns null, missing IDs, malformed numeric strings, null users, non-application user objects, and the intended unauthenticated/programmer-error path. Use distinct exception messages for missing IDs, wrong raw types, and non-positive or malformed values when that helps logs identify the broken invariant, and comment any production-impossible null raw-ID branch so it reads as a programming guard rather than an auth challenge. After replacing casts with programming-error guards, inspect nearby `try/catch` blocks so a broad `Throwable` catch does not swallow or remap `LogicException` into a client validation/domain error.
- When enforcing per-user caps or queue limits on soft-deletable models, state whether trashed rows count. Eloquent's default scope excludes soft-deleted records from `count()`, which may be correct, but the active-only versus lifetime-cap decision should be visible near the query and covered when it affects client-visible conflicts.
- When a controller catch or action guard is dead for normal HTTP callers because the FormRequest already rejects the input, leave a short comment that it protects direct action callers and add direct action coverage for that defensive path.
- Keep sibling domain exception hierarchy consistent. If conflict and validation exceptions use meaningful base classes, avoid introducing a not-found or input exception as a bare `RuntimeException` unless the difference is intentional and controller/action callers are prepared for it.
- Prefer typed FormRequest accessors for validated controller inputs when a request already exposes them. Avoid mixing `$request->validated()` array indexing for one field with accessors for the rest unless the asymmetry is intentional. When a controller calls custom accessors supplied by a trait, confirm they are visible in the touched trait/request, have a clear return type or PHPDoc shape, and return the payload form the controller expects.
- For PATCH pairs where one field describes another, such as `previewAudioRole` and `previewAudio`, define the invariant against existing model state. Either allow role-only updates intentionally and comment the partial-update semantics, or add a cross-field/state-aware guard so a role cannot exist without the corresponding media reference. Cover same-request clear-plus-role cases, not only role-without-existing-media cases.
- Audit cross-domain imports in new models/enums/actions. A Study class depending on Flashcards types can be correct, but call out the intended layer direction in the PR when the dependency could otherwise become a future cycle; keep the translation in one obvious location instead of spreading the dependency through the domain. When the dependency is only for persistence, consider storing a scalar at the boundary and hydrating the other domain's enum closer to that domain.
- When one domain maps to another domain's enum with `match`, add an exhaustiveness test or follow-up reminder so new/renamed cases in the source enum cannot create a hidden runtime unmatched-arm failure.
- When status/lifecycle columns imply required payload fields, make the assumption explicit near the model, migration, or factory state. For example, a `generating` draft with non-null seed JSON should say whether generation starts from user-supplied seed content or should allow empty pending content.
- Name lifecycle helpers after the actual state transition they perform. Avoid terms like `pending` when the enum/status written is `Generating`; prefer names such as `resetForRetry()` or `markRetryGenerating()` that match the persisted status and user-visible behavior.
- When a retry/reset endpoint moves a failed record back to `generating`, `pending`, or another worker-owned state, prove or document how work resumes. Either enqueue/dispatch the worker in the action, rely on and test an existing scanner/poller, or leave an explicit follow-up/client-timeout contract so API polling cannot wait forever on a state nobody advances. Also decide the dropped-response retry contract: if a successful retry changes state and a repeated `POST` now returns 409, mobile/offline clients need either an idempotent "already retried" response shape or documented logic that treats the conflict as an ambiguous transport retry instead of a hard failure. For idempotent already-in-target-state branches, maintain the same cleanup/invariant as the main transition when stale data is possible, or comment why in-flight records are intentionally returned untouched. If the branch clears worker-owned fields while the record is already in a worker-owned status, confirm the worker writes outputs and terminal status atomically or document why a concurrent transport retry cannot erase in-flight output.
- Result objects and action return values should expose only reachable information. If every successful return sets a boolean such as `shouldProcess` to the same value and failures throw before returning, remove the field, always perform the side effect, or add a real no-op path; dead result fields make controllers look conditional when they are not. If an action/bridge returns a model such as `SyncFeedEntry` and every call site discards it, prefer `void` unless sibling action consistency or an expected near-term caller justifies the return contract.
- Keep factory states that model public/media payload shapes synchronized with the canonical resource shape once that resource exists. Hardcoded draft/fake shapes are fine for storage slices, but should not silently become a second contract.
- Resource serializers should default to the model's cast contract via magic properties or `getAttribute()`, especially for client-facing enum values. Use `getAttributes()` only when raw storage values are the explicit contract, and test that choice; it returns database scalars and bypasses enum casts.
- When a helper only needs a backed enum's `->value`, type the input as `\BackedEnum|null` rather than a growing union of specific enum classes. Match the return type to `BackedEnum::value` (`string|int|null`) or cast/guard explicitly when the wire contract must be string-only; specific unions are useful only when the helper depends on those enum identities, all accepted enums are intentionally string-backed, or per-enum behavior is required.
- Resource helpers that coerce values should state and enforce their value-shape contract. If a helper is string-only for enum/string fields, document that or guard unexpected booleans, integers, arrays, and objects so future fields do not silently serialize with the wrong JSON type.

## Compatibility Adapter Invariants

- Keep compatibility controllers thin: translate request/response shape at the edge and reuse the canonical domain action for business behavior.
- Keep ConvoLab-specific field names and camelCase mapping in Study request/resource/support classes. Do not push those names into shared Flashcards, Reviews, Media, Courses, or Sync actions unless the canonical contract is intentionally changing.
- Test both the compatibility shape and the canonical invariant it delegates to: idempotent retry, no-op update behavior, sync feed emission or suppression, hidden 404 ownership behavior, and deleted-resource behavior.
- When a compatibility alias is intended to return the same response shape as the canonical route, assert the canonical load-bearing fields in the alias test too, not just a representative subset. If the canonical test checks fields such as `committedCardId`, sync metadata, or derived IDs, the alias test should either assert them or document why that field is intentionally omitted.
- When canonical endpoint coverage lives inside a compatibility test because the two routes share one bucket or invariant, make that discoverable: add a direct canonical test, a small comment in the canonical test file, or a PR note naming the compatibility test that owns the assertion.
- For idempotent retry/conflict behavior, test both the compatible no-op case and a true mismatch that must raise the conflict. This is especially important for canonicalized JSON, enum/type fields, and legacy-null defaults where the happy path can hide a broken comparison branch.
- For cross-path idempotency between canonical and compatibility aliases, cover both directions when the route pair is high-stakes for mobile/offline sync: alias-first then canonical retry, and canonical-first then alias retry. Keep mirror-direction tests symmetric in assertion depth too; if one direction verifies follow-up readback fields such as `committedCardId`, the other should either verify the same invariant or document why it is redundant. Cover mismatch/conflict symmetry too, such as alias-first commit followed by canonical retry with a different client resource ID and the inverse canonical-first/alias-different-ID case; if one direction is omitted because both routes share the exact controller/action, mention that assumption in the PR note.
- For hard-delete endpoints without tombstones, decide the retry contract explicitly. If already-deleted resources return 404 because ownership can no longer be proven, document that mobile/offline clients must treat that 404 as acceptable or add a tombstone/deleted-ID mechanism before promising idempotent 204 retries.
- When a compatibility response remaps resource data, prefer direct mapping from the canonical model/result over fragile round-trips through another resource's serialized array.
- When a response ID changes from echoing caller input to returning a canonical stored value, lock the compatibility contract for every identity shape the endpoint supports. Sourced integer-string IDs and unsourced ULID strings can both be intentional, but tests should make the difference visible.
- Keep HTTP/form field naming in the adapter layer unless there is a deliberate, narrow reason for a domain exception to expose it. If a domain exception returns a validation field name for controller mapping, treat it as an adapter convenience and keep those factories aligned with FormRequest `messages()` keys as new cases are added. For virtual error keys that clients never submit, such as aggregate `payloads` errors, align with sibling endpoints and add a test/comment so readers know the key is intentional.
- Do not make one FormRequest class the public helper surface for another request. If store/update requests need the same message builder, enum list, or payload helper, move it to a shared trait/support class that both requests already depend on.
- If a query scope joins tables or changes projection, assert that selected columns and loaded models still contain the expected base model fields. Watch for `select cards.*` or join projections wiping out data needed by a caller.
- Create-style compatibility endpoints that clients may retry need a stable client-supplied resource ID or an explicit "not retry-safe" contract. Idempotent setup work, such as resolving a default deck, does not make the final resource create idempotent by itself.
- Commit/convert endpoints need an explicit idempotency key decision. If a draft/import/job can be committed into a target resource, decide whether retry safety is keyed by the source, the client-supplied target ID, or a persisted source-to-target mapping; test or comment what happens when the same source is committed again with a different target ID.
- If a retry-safe commit stores the target ID back on a soft-deletable source, query with `withTrashed()` when retries after source deletion should still return the existing target. Otherwise a lost-response retry can degrade from idempotent success to 404; document the client contract if clients must keep the source until commit is confirmed.
- When a source gains committed/converted state but keeps its original lifecycle status, expose the new state or target ID in list/show resources, or explicitly document that clients must discover it another way. Hidden committed state can leave clients showing stale "commit" actions.
- For commit paths that write retry metadata after calling another action, decide what happens when the nested action throws. If rollback leaves the source marker null on conflicts such as deleted target IDs, test that state and document whether the client must choose a fresh ID.
- If a compatibility discriminator must live on a shared model, document the boundary decision, add lookup indexes for the real query shape, and test that unrelated canonical update APIs cannot flip the discriminator accidentally.

## Upload and Header Validation

- FormRequests own HTTP header syntax and should report malformed headers through Laravel validation errors, not domain exceptions. Domain actions own semantic checks and direct-caller defense.
- Persist and compare authoritative values from actual received bytes. Caller-declared sizes such as `Content-Length` are inputs to validate, not source-of-truth fields to store.
- Count binary payload bytes with `mb_strlen($contents, '8bit')` when file/archive content is involved.
- Cover all load-bearing upload guards: empty content, actual bytes over max, declared size over max with small actual payload, declared/actual mismatch, malformed header, native-integer overflow with longer digit count, equal-length value just above `PHP_INT_MAX`, and success with matching declared size.
- For failure paths, assert no DB mutation and no storage write when validation happens before persistence.

## Concurrency and Transaction Boundaries

- Expire-then-check, claim-then-process, and active-record guards should run under the same transaction and per-user or per-resource lock pattern as the write path they protect.
- Normalize cheap scalar inputs such as ULIDs before entering a transaction/lock when the normalization does not depend on locked state. This keeps failure paths outside the lock window and makes captured canonical values consistent.
- For update/autosave endpoints that first resolve a model in the request and later reject mutable states in the action, review the stale-read window. If a background job can flip status between authorization and save, either lock/refresh inside the write transaction or document that the next retry safely observes the conflict.
- Name stale-read/lock tests after the invariant they actually prove. A test that passes a stale model but expects the action to read the locked row should say that; a test whose action only accepts IDs should say it reads current DB status rather than a creation-time or pre-mutation snapshot. Avoid names that imply an impossible preloaded-model path.
- Move validation that can be decided from request data plus already-authorized model state before `lockForUpdate()`. A 422 that does not need a fresh locked row should not acquire a write lock; keep locked-action checks for invariants that truly depend on the current committed row.
- When different lifecycle branches deliberately clear versus preserve generated output fields, put the ownership rule near the save/transition that relies on it. Comments should say which worker/action owns in-flight outputs and why a success, retry, or already-generating path preserves them.
- If an empty PATCH/no-op path skips the normal write transaction, review it as a separate race path. It should still re-check ownership, soft deletion, and mutable status when those outcomes are client-visible, with focused action coverage if the branch has its own query.
- Defensive re-queries outside the main write path should state the race they protect, such as a delete between `authorize()` and action handling, so they do not look like redundant ownership checks.
- Durable behavior comments belong near the invariant owner. Prefer an action docblock or domain method comment for retry/idempotency/delete semantics rather than burying the only explanation in route definitions.
- If a read endpoint performs state mutation such as expiring stale jobs, review it like a write: transaction boundary, lock ordering, cross-user isolation, and race behavior.
- Avoid nested transaction surprises around actions that already require an outer transaction. If an action must be called inside a transaction, assert the transaction-level contract or fail fast with a clear exception.
- Do not wrap an action that owns its transaction in a controller `DB::transaction()` solely to make `DB::afterCommit()` available. Prefer placing `afterCommit` inside the transaction-owning action or dispatching plainly after the action has committed; if an outer transaction is intentional, comment the caller contract and savepoint cost.
- For double-check locking patterns, test both first-create and race/adopt paths, including soft-deleted rows that must not be resurrected or adopted accidentally.
- Keep side effects such as storage writes, sync feed entries, and job state transitions after all validation and conflict checks that can fail.
- Capture stable timestamps before retryable `DB::transaction()` closures when every attempted write should share the same logical time. Calling `now()` inside the closure can drift if the database layer retries after a deadlock or serialization failure, and can make frozen-time tests fragile when future closure work shifts evaluation order.
- Error/reset paths for paired generated fields should clear the full pair. If a model has both `preview_audio_json`/`preview_audio_role` or analogous image/media value-plus-role fields, clearing only the value can leave stale metadata attached to an error state.
- When persisting retry/idempotency metadata on a model with observers or sync-feed events, choose `save()` versus `saveQuietly()` deliberately. Leave `save()` when clients should learn the new state through normal model events; use `saveQuietly()` or comment the suppression when the marker is internal.
- Server-derived fields should not produce user-visible updates merely because a legacy nullable row is being backfilled in memory. If a nullable migration window exists, test that unchanged user content does not emit sync/feed entries only to fill a derived column.
- When a dirty-check guard exists to skip a defensive save, cover both sides when the save/no-save distinction matters: one case where stale state is repaired and one clean steady-state case that proves the branch does not issue an unnecessary write.
- Treat mid-request deletion of users, decks, cards, jobs, or other locked rows as a runtime race, not a programming error. Prefer a domain-appropriate 404/403/409-style exception over `LogicException` or another path that surfaces as a 500.
- When a shared or exclusive lock protects a later action only because the caller keeps an outer transaction open, make that caller contract explicit in the action docblock and tests. Locks acquired inside Laravel nested transactions/savepoints are held to the outer commit on MySQL/Postgres, but that dependency is easy to miss.
- Keep steady-state paths from taking broader locks than the invariant needs. A parent/user `lockForUpdate()` may be right for first-use creation, but existing-resource paths should usually prefer a narrower resource lock, shared lock, or no lock when the later write is already protected.
- Choose `sharedLock()` versus `lockForUpdate()` intentionally and comment the concurrency tradeoff when it is not obvious: shared locks can block concurrent updates/deletes while permitting concurrent reads/creates, whereas exclusive locks serialize more than necessary.
- Remember SQLite does not exercise row-lock semantics like MySQL/Postgres. If correctness depends on row locks, pair SQLite feature tests with transaction-level assertions, grammar/SQL checks, or narrow comments explaining what local tests cannot prove.

## Queued Jobs and Worker Boundaries

- When controllers dispatch queued jobs for a specific model/resource, decide whether duplicate jobs should be prevented with `ShouldBeUnique` or a similar unique key. A locked/idempotent action may make duplicate jobs safe, but queued duplicates still add avoidable work and lock contention; if duplicates are intentionally allowed, note why.
- Give new jobs an explicit retry envelope when failures can leave user-visible state pending: `tries`, `backoff`, timeout, or failure handling. Relying on Laravel/global defaults is acceptable only when the PR notes the inherited behavior and the state can safely wait for the next retry or manual repair.
- Match `backoff()` arrays to retry windows, not total attempts. For example, `$tries = 3` gives two retry delays; either return two values or raise the attempt count when a third delay is intended. Dead backoff entries are misleading failure-envelope documentation.
- If a request changes durable state into a worker-owned pending status and then dispatches a job, tie the dispatch to the same transaction/commit with `DB::afterCommit()` or an equivalent pattern. Dispatching after the transaction without a commit hook can leave records permanently pending when the queue write fails; retry/idempotency branches should either re-dispatch safely or avoid marking the state until dispatch is guaranteed.
- Jobs that own user-visible pending states need a final-exhaustion path, usually `failed(Throwable $e)`, that transitions the resource to an actionable error state or emits an equivalent monitoring/user signal. Keep it idempotent with missing/terminal/committed guards, use the same ID normalization contract as `handle()` or normalize once in the constructor, and clear any worker-owned output fields using the same pairwise cleanup rules as normal error transitions.
- Keep job lifecycle dependencies explicit, but respect queued serialization. If `handle()`, `failed()`, or another framework-invoked hook supports method injection or a constructor dependency that will not serialize services, prefer that. If the hook is called directly or constructor injection would serialize a service, defer container resolution inside the hook with a short why-comment; prefer direct `app(Foo::class)`/`app()->make(Foo::class)` unless `App::call()` is buying real method-injection value.
- Do not make Action state-transition internals public static merely so a job can call them from `failed()` without resolving the Action. That couples worker code to Action implementation details and bypasses future constructor dependencies. Prefer resolving the Action through the container inside the lifecycle hook, or extracting a small domain service/helper when the transition is intentionally container-free.
- In queued job methods, avoid non-static transaction closures that implicitly capture the whole job object. Copy only the scalar IDs/values needed into locals and use `static function () use (...)` when the closure does not need `$this`.
- When normal processing and `failed()` both write the same error state, keep the transition in one helper/domain method when practical. Duplicated status/error/output-clearing assignments drift as new preview/media fields are added.
- Avoid burying client-visible final-failure text in a private job body. Prefer a named constant, domain exception factory, or other domain-owned message source so retry/error strings are discoverable and consistent with nearby validation/domain errors.
- For `ShouldBeUnique` jobs, document the intended unique-lock behavior across attempts, backoff windows, and final failure for the Laravel version in use. Verify the deployed queue/cache backend provides the lock TTL semantics being relied on when `uniqueFor` is omitted; otherwise the uniqueness envelope may be weaker than comments imply and correctness must come entirely from processor guards. Keep comments aligned with the PR description and framework behavior that was actually checked; avoid precise lock-lifetime claims in code comments when the important invariant is simply that duplicate dispatches are harmless. Add or omit `uniqueFor` only when that lock window is intentional.
- If a queued side effect is passed into a domain action as an `afterCommit` callback, treat it as a local compromise. If the action already owns and commits its transaction before returning, consider direct post-action dispatch in the controller; use a small result object when the controller needs to distinguish a real transition from a no-op retry. If omitting the callback leaves a worker-owned state stuck, prefer making the callback required, emitting a domain event, or documenting in the action docblock that direct callers are responsible for driving the lifecycle forward. Also decide how an `afterCommit` callback exception should surface, because it can produce a client-visible 500 after durable state has already committed. Keep domain-action docblocks generic, such as "dispatch a processing job", instead of naming a concrete job class. If the pattern spreads across actions, prefer a domain event plus listener or another boundary that keeps queue concerns out of domain method signatures.
- Public helper methods that perform state transitions, such as `canProcess()` or `markAsFailed()`, should state at the method signature whether they require the caller to hold a row lock or transaction. Keep lock-sensitive helpers private when only one action owns them, or document and test the locked-caller contract when a job/failure callback must reuse them.
- Callback/closure parameter names should show whether an ID has been normalized or reloaded. Avoid shadowing an outer raw route/body variable with the same name when the callback receives a canonical ID.
- If dispatch is gated by Eloquent dirty tracking such as `wasChanged('status')`, cover the no-dispatch path with queue assertions and keep the action's return/save behavior stable enough that the controller is not guessing from stale in-memory state. When a controller needs to decide whether to enqueue, emit sync, or perform another side effect based on an action outcome, prefer a small domain-specific result object over inferring business outcomes from model dirty state.
- For worker actions that no-op terminal or committed resources, test the edge state the guard names, not only the common terminal status. Cover each terminal enum/status that has distinct meaning, such as `Ready` and `Error`, and include a `committed_card_id !== null` committed-but-still-generating/race fixture when that state is possible.
- Job tests should make payload intent explicit when validation success is the behavior under test. Avoid relying on factory defaults to happen to satisfy a worker's validator when a small override would make failures point at the real invariant.
- Queue assertions with exact counts should make the request count obvious. If a test posts twice and expects `assertPushed(..., 2)`, keep the structure or a short comment clear enough that future readers do not read the count as a stale assertion.
- Remember `Queue::fake()` proves a dispatch attempt, not production `ShouldBeUnique` lock acquisition or actual enqueue persistence. When uniqueness/backoff behavior has user-visible consequences, add a comment or integration-style coverage for what fake assertions cannot prove, such as an explicit retry being silently de-duped while the original job is still backing off.
- When comments rely on duplicate dispatches being harmless, cover the contract directly: either test duplicate jobs no-op through the processor's guards or add environment/integration coverage for the queue uniqueness backend where feasible.
- Split worker no-op tests when combined scenarios can mask each other. Missing resources, ready/error terminal statuses, and committed-race guards often deserve focused tests once lifecycle behavior starts growing.

## Existing Coverage Must Not Weaken

- Do not change an existing test from a stronger input to an easier one just to add another scenario. Add a new test or keep the old assertion.
- When replacing helper code, ensure all previous edge cases still have named tests.
- If a PR removes a guard, assertion, migration fixture, or branch, explicitly verify that another test still covers the invariant.

## Migration and Postgres Portability

Postgres compatibility is first-class for this project. Take migration portability comments seriously even if the current local database is SQLite.

- For schema or index changes, include grammar fixture tests for SQLite, MySQL, and Postgres when the generated SQL can differ by grammar.
- Assert rollback/drop SQL, not just create SQL.
- Pin exact compiled SQL only when the SQL text is the real risk, such as Postgres identifier length, dialect-specific clauses, or rollback grammar. Prefer structural schema/index assertions when a Laravel grammar upgrade could fail a full-snapshot test without a semantic schema change.
- Migration data backfills should state their operational shape when row counts may be large: chunk size, one-update-per-row behavior, expected runtime/lock impact, and why set-based SQL is not being used.
- Keep migrations self-contained even when that duplicates application helpers, but leave a short comment when duplicated parsing/normalization logic is intentional so future edits do not accidentally couple old migrations to evolving app code.
- Keep index names under Postgres's 63-byte identifier limit and test the actual names generated by the migration.
- Verify column order against the real query pattern. Put equality predicates before range/order columns when that is what the query uses. If a trailing column is included only to make an index covering for `COUNT(column)` or a selected field, document that tie to the current query shape; if the query later switches to `COUNT(*)` or stops selecting that field, the covering column may become dead width on Postgres/SQLite, while MySQL InnoDB may already append the primary key implicitly.
- For low-cardinality filters such as type/status, confirm the index starts with the most selective real predicate. A filter-only index may be dead weight when every query is user-scoped through another table; use `EXPLAIN` or a production-shaped query analysis when in doubt.
- When a new foreign-key/reference column is intentionally unindexed because all reads use another key, leave a short migration comment if grammar tests pin exact SQL. Otherwise future reviewers may read the missing index as an accidental omission. If a partial index such as `WHERE deleted_at IS NULL` would be better for the eventual Postgres-only target but the migration stays cross-database today, call out the portability tradeoff or follow-up so the full index does not look like an overlooked optimization.
- Use one source of truth for expected index names and fixture SQL where possible.
- If SQLite and Postgres expectations are intentionally identical, leave a concise comment so future maintainers know the duplication is deliberate.
- When migration tests intentionally duplicate a migration blueprint or DDL snapshot, add a concise comment pointing future editors back to the migration as the source of truth so schema changes are updated in both places.
- If a grammar fixture test uses a lightweight or mismatched PDO only to compile SQL for another dialect, keep the test strictly compile-only and comment that it must not exercise connection/PDO execution paths.
- Confirm the migration file actually exists, is loaded by the test, and matches the column/table names used by production queries.
- For enum-like state columns, decide whether the invariant lives at the database layer. Laravel `enum()` can compile to check-backed SQLite/Postgres SQL and native MySQL enum on create, but `change()` migrations can be SQLite-rewrite-heavy; do not introduce enum/check-constraint DDL unless SQLite, MySQL, Postgres, and rollback behavior are proven.
- Finite role/status/type strings should usually have a backed enum and Eloquent cast before public APIs depend on them. Leaving a role as a bare string invites invalid values and undocumented contracts.
- When a DB check constraint is deferred, keep the application serializer defensive around nullable legacy/raw rows and state the tradeoff in the PR note.

## Query, Cursor, and Ordering Invariants

- Cursor pagination and access token lists need deterministic ordering, including tie-breakers for equal timestamps.
- Cursor tokens and query binding must preserve the precision of every ordered column. The cursor value used in `>` and `=` keyset branches must round-trip at the same precision the DB stores; if the column is second-precision, encode/truncate to seconds deliberately, and if it is microsecond-precision, test page edges with non-zero microseconds.
- Cursor precision comments should explain the pagination consequence, not just the column type. State why truncation or exact precision is required for the equality tiebreaker branch to find same-timestamp rows.
- Cursor decoders should validate every required field before parsing or constructing domain values. Guard empty timestamp strings explicitly, and prefer exact-format parsing such as `createFromFormat()` for server-generated opaque tokens; permissive `Carbon::parse()` / `CarbonImmutable::parse()` can accept relative or partial strings that should not be valid cursors.
- Cursor encoders with persisted-model preconditions should have guard tests too. If `encode()` throws when `id` or `created_at` is missing, add a focused test for a new/unpersisted model.
- Define pagination metadata semantics clearly. If `total` is the full filtered count rather than remaining count after the cursor, keep it stable across pages and test or document the client-facing contract.
- Choose default and max page sizes with payload weight in mind. If each item can include large JSON blobs or media metadata, document the intended response-size tradeoff or a known safe bound instead of only validating the numeric limit.
- For list endpoints, include the empty result contract in either action or API coverage: empty items array, null next cursor, and the expected total/count semantics.
- For user-scoped cursor endpoints, test cursor reuse across users. A valid cursor produced for user A must not leak, skip into, or shape user B's result set beyond B's own scoped records.
- Avoid duplicate cursor decoding on the HTTP path when a FormRequest already validates and parses the cursor. Prefer a typed accessor/DTO from the request if the action can accept it without weakening direct-caller validation; if the action deliberately decodes again to protect direct callers, leave a short comment so HTTP 422 vs action exception boundaries stay clear.
- For case-insensitive ID compatibility without losing indexes, prefer precomputed exact candidates over SQL functions such as `LOWER(column)`, but assert or handle the "no preferred candidate matched" and "multiple case variants matched" paths explicitly. Fallbacks after a narrowed `whereIn` should return empty or fail loudly, not silently return the unfiltered collection. If the result is sorted or filtered in memory by candidate preference, keep the candidate set deliberately bounded and nearby in code/tests so the pattern is not copied onto broad lists.
- Normalize collection keys consistently after pagination slicing. Even if `get()` currently returns zero-based keys, call `values()` on all paginated branches when the response contract is a JSON array.
- When adding an index for a future keyset-paginated list, include the deterministic tiebreaker column such as `id` in the intended ordering/index plan, or explicitly defer it until the query layer lands.
- If adding or changing a filter predicate, confirm there is an index that supports the new query shape.
- Keep pagination query strings stable when filters are present; tests should confirm filters survive next/previous links.
- Avoid hidden assumptions such as integer primary key ordering unless that is part of the documented model contract.
- Helper methods that build query patterns should make blank rejection and wildcard escaping explicit, including `@throws` docs or direct tests when direct action callers can bypass HTTP validation.
- For facet/filter option endpoints, pin the cross-filter contract in tests: selected filters should remain visible when appropriate, unfiltered/search-only paths should return the same options as the SQL fallback, and query-count optimizations must preserve semantics.
- For aggregate subqueries and `leftJoinSub` refactors, test zero-count rows, cross-user isolation, deleted cards/decks, and the exact fields needed by detail endpoints. When replacing a shared scoped helper with raw SQL or a scalar subquery, enumerate the helper's predicates first and preserve every implicit scope, including soft-delete filters on each table, ownership joins, deck filters, status/active scopes, and intentionally user-wide versus deck-scoped dimensions; add fixtures for any predicate that is easy to omit, and assert the exact aggregate value those fixtures affect so they are load-bearing. Remember that `SUM()` over no matching rows can return `NULL`; make the zero contract explicit with SQL `COALESCE` or a documented cast. When metrics are similar but not identical, such as failed versus failed-and-due, include fixture rows that make the counts diverge. When one metric groups several enum/status values, seed at least one row from each grouped value unless another test already locks that member.
- Raw SQL aggregates with repeated `IN (?)` lists should derive placeholders from the same arrays used for bindings, for example with `array_fill(0, count($values), '?')`, or fail loudly with a guard when a fixed placeholder count is intentional. Do not let helper-list size changes break PDO bindings at runtime. For `DateTimeInterface` values in raw binding arrays, prefer explicit formatting when timezone/storage representation matters and normalize to the DB storage timezone before formatting if callers can pass non-UTC instances; for long positional binding arrays, group or comment bindings by the SQL expression they satisfy.
- When optimizing query count, assert the count against the behavior being optimized and keep a semantic regression test beside it; a faster wrong result is still wrong. Avoid whole-action query totals when the goal is one subquery/path unless the expected total is documented, because unrelated collaborators can add queries and fail the optimization test for the wrong reason.

## Sync and Offline Invariants

- Client-provided IDs should preserve idempotent retry behavior.
- Cross-user conflicts should stay hidden as 404/empty responses when that is the API contract.
- Deleted resources, tombstones, and conflict responses must remain consistent with existing sync behavior.
- Resource payload keys should remain client-facing and include context such as `course_id` consistently across equivalent sync resources.
- Enum-cast fields in resource or sync payloads should serialize the client-facing value, but include a default or explicit guard when null legacy/raw fixtures can reach the serializer.
- If a resource passes JSON-cast fields through directly, assert representative `prompt`/`answer` or media JSON payloads in API tests, not only surrounding scalar fields.
- For sync feeds, check ordering, checkpoint behavior, and payload shape together; Claude often flags only one symptom of a broken invariant.
- Sync feed entries should represent durable state changes, not every request-shaped attempt. When adding feed recording to create/update/delete/retry actions, cover transport retries, no-op same-value updates, already-deleted or already-terminal resources, and client-provided-ID create paths so `Create`, `Update`, and `Delete` entries are not duplicated. Use explicit guards such as conflict-before-record, `wasRecentlyCreated`, `isDirty()`, or an operation-specific no-op branch when that makes the invariant self-documenting.
- When a new side effect is added to an action, update existing data-provider tests for terminal, missing, already-processed, or no-op cases to assert the side effect stays absent. A single companion test with `assertSame(0, ...)`, `Queue::assertNothingPushed()`, or storage assertions does not prove every provider case kept the invariant.
- Also update standalone no-op tests, such as missing-resource or already-deleted branches, when sibling no-op paths gained side-effect absence assertions. Gaps are easy to miss when one data provider and one single-case test cover the same family of early returns.
- If sync feed recording happens inside the same `DB::transaction()` as the model mutation, decide whether feed insert failures should roll back the model state. Keep that atomicity when the feed is part of the durable contract; use an idempotent recorder, conflict handling, `DB::afterCommit()`, or an outbox-style pattern when feed failures are recoverable side effects.
- For job final-exhaustion or other "last chance to escape pending state" paths, prioritize persisting the terminal/error state over recording lower-priority sync/feed side effects. A feed insert failure inside the same rollback boundary can strand the resource in a non-terminal state with no retry left. Moving the feed write to `DB::afterCommit()` avoids that rollback, but the callback can still throw after state commits and orphan the resource from the feed; catch/report it, provide a retry/recovery path, or comment that propagation is intentionally fatal.
- When one lifecycle path intentionally diverges from sibling transaction or side-effect placement, comment why the asymmetry exists, not only what the code does. For example, a job-exhaustion path may use after-commit best-effort sync because the terminal error marker must survive sync outages, while ordinary lifecycle actions keep sync atomic with retryable state changes.
- For hard-deleted resources that emit tombstones, document whether payload `deleted_at` is synthetic/caller-supplied and whether `updated_at` remains the pre-delete model timestamp. If the payload always includes `deleted_at: null` on non-delete operations, make the consumer contract clear: `null` means "not deleted" and absence means a different/older payload shape. This avoids future clients or maintainers treating `updated_at < deleted_at` or null tombstone fields as accidental drift.
- When using Eloquent dirty tracking to decide whether to emit sync or dispatch work, capture `isDirty()`/`wasChanged()` at the point where the framework still reports the intended state. If the flag must be captured before `save()`, add a short comment because `save()` clears dirty state and a future move can silently suppress or duplicate side effects.

## Rate Limiting

- Keep named limiter policy out of `AppServiceProvider` when the keying/defaults belong to a domain. Providers should wire the limiter; domain support classes should own the key formula and default limit.
- In `AppServiceProvider`, match the registration style already used nearby. If existing named limiters instantiate domain limiter objects before the `RateLimiter::for()` closure and capture them with `use`, follow that pattern for new sibling limiters unless the PR explains a deliberate convention change; otherwise equivalent inline factories can look like accidental drift.
- When a domain adds several sibling limiter classes, choose the abstraction based on the duplication that exists now. If only the key formula repeats across several limiters, centralize it in a small domain helper that accepts the limiter name; a second small copy may be acceptable in a narrow PR, but adding a third byte-identical key helper should trigger extraction or a clear follow-up note. If the whole class differs only by `NAME` and `PER_MINUTE`, prefer one parameterized domain limiter with named constants over three structurally identical classes. Name operation-specific factories as selectors such as `forCreate()`, `forUpdate()`, and `forDelete()`; a bare `create()` reads like a generic constructor alias rather than "limiter for create." Do not duplicate `user`/`anon`/`unknown-ip` key logic across create, update, and delete limiters.
- Use one source of truth for limiter names across `RateLimiter::for()`, route middleware, tests, comments, and PR descriptions. Do not describe a limiter as `*-write` while the constant is `*-create`; clients and tests may observe names in throttle headers or route middleware strings, so naming drift is review-worthy even when behavior is correct.
- Do not assume whether distinct named limiters share or isolate Laravel cache buckets. Check the pinned framework path: in Laravel v13 named throttle middleware prefixes the limiter name before hashing, while direct `RateLimiter` API calls or custom middleware may use only the raw key. If separate quotas are part of the contract, either rely on and cite/test the named-middleware prefix behavior or include an action/limiter prefix such as `study-card-update:user:{id}` in the raw `Limit::by()` key; add a cross-limiter assertion or PR note when the distinction is review-worthy.
- When multiple endpoints reuse the same named limiter, state whether they intentionally share one quota bucket. If the interaction is user-visible, add cross-endpoint tests or PR notes that cover every route alias in the shared group, not only one representative pair; a strong test consumes quota through one route and asserts a sibling route with the same limiter is blocked. If the self-review claims compatibility or sibling routes were already wired in an earlier PR but the diff does not show them, cite the prior PR or include a grep/route reference so reviewers can verify the shared bucket exists. Check starvation between operation classes before sharing: bulk/offline replay creates, bulk token revokes, or sign-out-all-devices flows can exhaust a bucket and block reactive undo/delete/current-token actions that users expect to work immediately. Place route comments so they visibly cover every route sharing the quota and explain the chosen ceiling when the number is not obvious, not only the route immediately below the comment, and make PR descriptions match the actual partitioning rather than implying separate create and undo buckets when the implementation uses one. If shared-route coverage lives in only one sibling test file, mention the cross-route assertion in the PR note or add a small pointer so reviewers do not think the unmodified sibling test file lacks coverage.
- High-frequency write endpoints such as autosave should have an explicit throttle decision. Add permissive per-user throttling, reuse a documented shared bucket, or explain why the endpoint is safe without route middleware; do not leave rapid PATCH/retry paths unbounded by accident.
- For batch write endpoints, decide whether the limiter controls requests or items. A request-based limit on `POST /batch` can allow `limit * max_batch_size` writes per minute; confirm the batch-size cap exists and call out the effective per-item ceiling in the PR note or route/domain comment when that is intentional for offline replay. If per-item volume matters, use per-item accounting inside the action or a separate lower batch limit.
- For offline/retryable create endpoints, remember route throttle middleware runs before action-level idempotency checks. A client reconnecting with a large backlog or retrying lost responses can hit `429` before the create action de-duplicates; choose the quota with that ordering in mind and preserve the acceptable backlog/retry tradeoff near the limiter registration, domain constant, or route comment instead of only in the PR description.
- Destructive endpoints such as DELETE are often manual gestures, but client bugs and retry loops can still hammer them. If sibling create/update routes have throttle middleware, default to adding a generous route limiter to DELETE too; rely on global API throttling only when that is an established repo convention and the PR calls it out.
- When throttling retry-safe soft-delete endpoints, check both retry windows. A DELETE that succeeded but whose response was lost can be retried after the throttle window when the model is already trashed; the HTTP layer should use `withTrashed()` or equivalent when 204 retries are promised, and the action should treat already-trashed models as successful no-ops. A retry during quota exhaustion may still get `429` before the idempotent path runs, so decide whether that unresolved sync state is accepted and documented, needs a higher/separate delete quota, or needs an exempt already-deleted fast path before throttle middleware; keep focused tests or cite existing coverage for the chosen contract.
- Make rate-limit constants self-contained. A comment like `matches saved-card edit headroom` should include the actual quota and reason, or name the sibling constant in code, so future readers can verify parity without opening another file. Do not copy a sibling quota blindly: compare normal endpoint frequency, sync retry policy, retry-storm blast radius, and operation sensitivity, especially for lower-frequency settings/profile updates and credential changes. For password or token-security writes, use a tighter quota than ordinary profile edits unless the PR calls out a product/retry reason for the higher limit.
- Test key fallbacks for missing user/email, empty strings, whitespace-only values, non-string payloads, missing or empty IP, and sentinel text used as a real value. Cover the mixed quadrants too, such as a valid identifier with missing IP and a missing identifier with valid IP, so each fallback segment is proven independently. Expose a public `keyFor()`-style method only when direct fallback cases cannot be asserted through `limit()` or an HTTP path; otherwise keep key construction private. If a key method is public only as a testable contract, mark it `@internal` or comment that route code should use the named limiter methods. Match key-helper parameter types and normalization branches to the app's actual guard contract and direct peers; do not narrow one sibling while peers expose a wider contract, but when there is no peer constraint and production only calls with `int|string|null`, prefer that signature over `mixed` plus silent boolean/object fallback branches. Do not add object/`__toString` branches unless a real custom guard or direct caller can hit them. If booleans/falsy values are possible in tests, cover `0`, `false`, and `true` separately or narrow the signature so those impossible values cannot be passed; if `0` or `'0'` could mean guest/unauthenticated for the guard, either fall back to IP or comment that zero values are intentionally valid user identifiers. If the app requires positive numeric IDs, decide whether integer zero, string zero, negative integers, numeric-looking negative strings such as `'-1'`, and decimal strings such as `'1.5'` should be user-keyed, rejected, or treated as fallback and cover that contract. Do not let a general non-empty string branch accidentally accept invalid numeric strings while rejecting only invalid integers. If the string path is defensive but production IDs are integers, simplify the production guard or extract a small `isValidUserId()` predicate rather than leaving dense inline boolean logic. When testing ordinary non-null string IDs, use neutral values like `str-id` or a sample ULID rather than placeholder text such as `missing-user` that reads like a fallback scenario.
- Choose rate-limit key dimensions deliberately. Combining user ID and IP gives the same user fresh buckets when networks change; if the policy is truly per-user, use user ID when present and reserve IP only as an unauthenticated fallback, then test/comment the chosen behavior.
- Each distinct limiter bucket should have user-isolation coverage for the operation class it protects. Do not rely on create-path user isolation to prove undo/delete isolation when those paths use a separate limiter constant; if a broken key fell back to IP, same-IP CI users could hide the regression.
- For auth-gated routes with an anonymous fallback such as `anon:unknown-ip`, decide whether that branch is unreachable defensive code, a real anonymous bucket, or should fail closed. If middleware order could let throttle run before auth, a generous shared unknown-IP bucket can silently block unrelated anonymous callers; use a tighter anonymous quota, a clearer comment, or a direct auth-before-throttle assertion when the distinction matters.
- Avoid sentinel collisions in limiter keys. Prefix typed identities such as `user:123` and `anon:1.2.3.4` instead of using a bare value like `missing-user` as both a placeholder and a branch condition.
- When changing an existing limiter key format, call out the rollout effect: old cache buckets become orphaned and users mid-window may receive a fresh quota. For low-risk creation limits a PR/deploy note may be enough; for stricter abuse controls, plan migration, dual-read, or temporary monitoring.
- When tests temporarily override a named limiter, isolate hit counters with a unique per-run bucket or clear the exact keys when cleanup is load-bearing; do not add redundant pre-clears for freshly generated buckets, but keep final cleanup when a persistent cache would otherwise accumulate orphaned counters/timers. Restoring only the limiter definition is not enough because cached attempts can leak. `RateLimiter::for()` overrides also mutate process-global handler state, so a cache-key prefix does not make the test parallel-safe. Put the override itself inside the `try` block protected by the `finally` restore; do not leave a throwing gap between global mutation and cleanup registration. If a test grows setup, callbacks, or helper calls around the override, prefer a dedicated wrapper helper or `setUp`/`tearDown` pair so no future assertion accidentally runs under the overridden process-global limiter. Prefer direct `Limit` unit coverage, a test-specific closure around the production key helper, a fake-cache integration shape, or a uniquely named test limiter over replacing the production limiter name. If a feature test must override the production name, verify the project's PHPUnit/Pest/GitHub Actions parallel runner actually excludes the chosen group using syntax recognized by the pinned test framework, such as `#[Group('no-parallel')]` versus legacy docblock annotations where applicable, or uses another documented isolation mechanism; otherwise the annotation is only documentation. Do not add future-only group markers that look protective but are not wired to anything; either wire the exclusion, omit the marker, or label it clearly as unenforced. Fall back to a comment only when CI/test docs prove the suite is never run in parallel.
- Assert the throttled response status, default limit values, 429 body contract, and at least one representative 429 header path when a PR changes limiter behavior or its tests. Offline clients may use `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `Retry-After` for backoff, so do not only assert the status code. Header presence is acceptable for broad smoke coverage, but when a test installs a fixed `Limit::perMinute()` override with the default 60-second decay, prefer `assertHeader('Retry-After', '60')` on the representative path so decay drift is caught. Laravel's default throttle body may be `{"message":"Too Many Attempts."}` rather than the app's normal API envelope, so either test/accept that divergent shape or add the render override expected by clients.
- Do not add public helper parameters such as `$perMinute` solely for tests when production always uses the default. Keep test-only limits inside the test's limiter closure, a dedicated test double, or explicit configuration; if a production override is real, cover it with a focused assertion and document the caller.
- Rate-limit tests should consume buckets with payloads and lifecycle state that are valid for the behavior under test, or clearly comment why an invalid/missing payload/state is safe. Otherwise the throttle assertion can become coupled to validation ordering rather than limiter behavior. For multi-step resources such as uploads/imports, seed the same completed/uploaded state the action contract requires; if the action still accepts an incomplete state, verify whether that is intentional or a pre-existing guard gap.
- Throttle test setup should not consume the production limiter before the test override is installed. If the scenario needs existing rows, seed them with factories or call the domain action directly; if HTTP setup is necessary, install the isolated test limiter first or explain why the real limiter cannot affect the test.
- When cleanup keys are constructed with simplified inputs such as `keyFor($userId, null)` but the request-time limiter uses `$request->ip()`, either pass the same request IP or comment that the authenticated branch ignores IP. If a test helper accepts a custom IP, cleanup must use that actual IP rather than a fixed default. If the limiter has an anonymous/IP fallback and tests can ever exercise it, clear that fallback key too; if auth middleware makes the fallback unreachable, comment that assumption near the cleanup. Avoid caller-managed cleanup lists when a helper callback can create additional users or identities; either record every key as the helper builds it, clear the whole unique test namespace when the cache API supports it, or keep the list API only for tightly scoped helpers whose callback cannot introduce new identities. This keeps final cleanup understandable when key helper behavior changes later.
- When a test-local limiter prefixes or otherwise transforms production keys, derive cleanup keys from the same helper/closure used to build `Limit::by()` or from `$limiter->limit($request)->key` on a synthetic request. Avoid hand-assembling prefixed keys in several places, and clear only buckets that could actually have been hit.
- When three sibling feature tests copy the same rate-limit scaffold, such as unique test bucket creation, IP setup, `RateLimiter::for()` restore closure, cleanup key computation, and `try/finally`, extract a small test helper or trait before adding a fourth bucket. The helper should still make the operation-specific payload and side-effect assertion visible in each test. Keep the helper API single-source: if every call passes a limiter name together with its matching default factory, derive the restore limiter from the name inside the helper instead of coupling every call site, but validate unknown names before entering the protected callback so a restore exception in `finally` does not mask the real test failure. Use precise PHPDoc for helper inputs, such as `list<int|string>` for sequential user ID arrays, rather than `mixed` when tests never pass arbitrary values. If several helper methods dispatch on the same limiter-name set, prefer one shared resolver/map or use a consistent exhaustive `match` that returns a class/value and then perform side effects, instead of duplicating match tables or mixing `match` in one method with side-effect-only `switch` in another. Put global test state changes such as `withServerVariables()` inside the protected `try` block or restore the prior value so the helper stays re-entrant within one test method. When setup uses a framework helper, restore through the matching helper, for example `withServerVariables($previousServerVariables)`, rather than assigning framework-owned properties directly.
- When a throttle test relies on pre-existing state to avoid side effects, such as pre-deleting records so allowed attempts are idempotent no-ops, comment that setup near the fixture mutation. When the assertion is meant to prove throttling suppresses writes, make the allowed attempts use a payload that would otherwise create the side effect, or explicitly state that the test only covers idempotent no-op attempts. Cover every side effect the unblocked endpoint normally emits: persisted model changes, soft deletes, review/status rows, and sync feed/event rows. Prefer actor/resource-scoped `assertDatabaseMissing` assertions over whole-table zero-count checks unless the empty table is itself the invariant.
- Be deliberate about whether throttle middleware runs before validation. If malformed requests consume quota, note that choice when clients may auto-retry validation errors; if only successful writes should count, the limiter must move after validation or into the action.
- When throttling only one verb in a shared route area, consider a small assertion that sibling reads remain unthrottled after the write quota is exhausted. This documents route-scope intent and catches accidental group-level throttle refactors.
- Avoid hand-computing Laravel throttle storage keys in tests. A helper that guesses `md5($limiterName.$key)`, `sha1($key)`, or another cache address can be a misleading no-op if it diverges from the actual middleware path. Prefer `RateLimiter` API calls with the same raw key the limiter returns, unique per-run key prefixes that make cleanup best-effort, or a focused assertion that proves the helper matches the framework behavior. If a test relies on cleanup rather than unique keys, verify the pinned Laravel version's `RateLimiter::clear()` removes both the counter and timer keys.

## Style and Maintainability Nits That Recur

- Avoid no-op `merge([])` calls or comments that describe a temporary implementation detail.
- Remove or annotate unreachable defensive catches, fallbacks, and branches that make normal data look more dangerous than it is. If a `JsonException`, domain exception, empty fallback, or other guard exists only for impossible values through the real query/request path, either prove the direct-caller path can reach it with a focused test/comment, make invariant violations loud, or simplify the code.
- Do not duplicate private helper methods across sibling tests or request classes when a local helper/trait already exists.
- Prefer static helpers for fixture/index-name builders that do not depend on instance state.
- Mark small domain support classes `final` when they are not extension points. `@internal` on one public test helper documents intent, but it does not prevent subclassing.
- Remove redundant casts, `forceFill` calls, or assignments unless they protect a real invariant.
- When embedding a Laravel resource or resource collection inside a custom response shape, prefer an approach that preserves the intended Laravel resource pipeline, and test the exact envelope. Inline `AnonymousResourceCollection` values can serialize cleanly inside `response()->json(...)` without a top-level `data` wrapper, but that is non-obvious enough to cover with tests or comments; direct `toArray()`/`resolve()` calls should be deliberate because they may bypass hooks, metadata, wrappers, wrapping, `additional()` data, or filtering behavior. For show endpoints, compare adjacent controllers before choosing `return Resource::make(...)` versus `response()->json(Resource::make(...)->resolve($request))`.
- If a short helper centralizes repeated guard and assignment logic, use it consistently across the touched fields.
- When extracting shared traits/helpers, preserve existing PHPDoc generic shapes and static-analysis contracts unless widening them is intentional and covered by callers. A runtime-equivalent type widening can still be a regression for tooling.
- If a recursive validator is bounded by a payload-size cap, keep that cap nearby in tests or comments so future increases do not turn an O(total nodes) walk into an accidental performance issue.
- Avoid creating thousands of Eloquent factory records just to reach a max-count guard. If tests only need aggregate count, use chunked bulk inserts, a small injectable/overridden limit, or a direct setup helper so the test proves the invariant without making every CI run pay for production-scale fixtures. Do not replace slow factories with one huge `insert()` that can exceed SQLite's bind-parameter limit; choose chunk sizes from the actual row width and leave margin for future columns instead of sitting at SQLite's 999-parameter edge. If the chosen chunk size is a non-obvious value like 90, comment the verified bind-limit arithmetic near the helper, such as rows times the real inserted column count versus SQLite's 999-parameter ceiling; incorrect arithmetic is worse than no comment.
- Match local Laravel model conventions unless there is a reason to introduce a newer mechanism. For example, use `#[Fillable]` only when the repo already does or the PR explains the move from `$fillable`.
- Do not add convenience helpers such as enum `values()` methods speculatively when local validation uses `Rule::enum()` or enum cases directly. Add them when a caller needs them, or note the intended follow-up use.
- Keep factories in the repository's established namespace/location for domain models; if a new domain suggests a different factory organization, call that out instead of letting conventions split silently.
